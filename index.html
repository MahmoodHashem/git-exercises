Deeper Look at Git

Changing History

  1. Changing the last commit

    Use git commit --amend
    Remember to only amend commits that have not been pushed anywhere! The reason for this is that git commit --amend does not edit the last commit, it replaces that commit with an entirely new one.

  2. Changing multiple commits

    This is where the beautiful command git rebase comes into play!

    Starting an Interactive Rebase

    • Initiate the Rebase:

      • To start an interactive rebase, use the following command:

      git rebase -i HEAD~N

      Replace N with the number of commits you want to affect. For example, if you want to modify the last 3 commits, use HEAD~3.

    • Understanding the Interactive Rebase Editor

      When you run the command, Git will open your default text editor with a list of commits, similar to this:

      pick abc123 Commit message 1 pick def456 Commit message 2 pick ghi789 Commit message 3
    • Commands You Can Use

      In the editor, you can replace the word pick with several commands:

      1. pick:Use the commit as is.
      2. reword:Use the commit but change its commit message.
      3. edit: Pause the rebase after this commit so you can modify it (e.g., change the content).
      4. squash: Combine this commit with the previous commit. You’ll be prompted to edit the commit message for the combined commit.
      5. fixup: Similar to squash, but it discards the commit message of the squashed commit.
      6. drop:Remove the commit entirely.
  3. Splitting up a commit

    git reset is a powerful command in Git that allows you to undo changes by resetting the current HEAD to a specified state. It can modify the staging area and working directory based on the reset options you choose.

    Types of Git reset

      1. Soft Reset (--soft)

      Moves the HEAD pointer to a specified commit but leaves your working directory and staging area (index) intact.

      This is useful if you want to uncommit changes but keep them staged for a new commit. 

      git reset --soft <commit>

      Example:

      git reset --soft HEAD~1

      This command undoes the last commit, keeping the changes staged.

      2. Mixed Reset (default)

      Moves the HEAD pointer to a specified commit and resets the staging area to match, but leaves the working directory unchanged.

      This is useful if you want to unstage changes while keeping them in your working directory. 

      git reset <commit>

      Example:

      git reset HEAD~1

      This command undoes the last commit and unstages the changes, but the changes remain in your working directory.

      3. Hard Reset (--hard)

      Moves the HEAD pointer to a specified commit and resets both the staging area and working directory to match that commit.

      This is useful if you want to discard all changes since that commit. 

      git reset --hard <commit>

      Example:

      git reset --hard HEAD~1

      This command undoes the last commit and discards all changes in the working directory and staging area.

      Use Cases for Git Reset

      • Undoing Commits: If you mistakenly committed changes, you can use git reset to undo the commit while deciding what to do next.
      • Unstaging Changes: If you accidentally staged changes you didn’t want to commit, a mixed reset will unstage them but keep them available for further edits.
      • Discarding Changes: If you want to completely remove changes made since a specific commit, a hard reset is appropriate.
